# MedKabinet

MedKabinet — информационная система для записи пациентов в медучреждение. Проект реализует базовые процессы клиники: ведение пациентов и врачей, просмотр расписания, создание записей на прием, самостоятельную запись пациента, работу с историей болезни, ролевую авторизацию, импорт и экспорт данных в CSV.

Система разрабатывается как учебно-практический проект с акцентом на предметно-ориентированную разработку: бизнес-данные представлены доменными сущностями и value object, а доступ к данным разделен на repository, persister, service и query-слои.

## Основные возможности

- Авторизация пользователей с ролями `ADMIN`, `DOCTOR`, `PATIENT`.
- Ролевое разграничение доступа к страницам и операциям.
- Дашборд нагрузки по врачам и отделениям.
- Просмотр расписания врачей по дате и врачу.
- Просмотр списка врачей и карточки врача.
- Редактирование данных врача администратором.
- Просмотр списка пациентов с пагинацией.
- Карточка пациента с персональными данными, страховой информацией и историей болезни.
- Создание и редактирование пациента.
- Запись пациента на прием администратором или врачом.
- Самостоятельная запись пациента на прием через пациентский интерфейс.
- Просмотр врачом только тех данных, которые относятся к нему.
- Добавление и редактирование истории болезни врачом.
- Экспорт карточки пациента и связанных записей в CSV.
- Импорт данных пациента из CSV.
- Экспорт и импорт истории болезни пациента в CSV.
- Хранение пользовательских сессий через Spring Session JDBC.
- Отображение ошибок на интерфейсе через Bootstrap toast и `invalid-feedback` у полей форм.
- HTMX-запросы для частичного обновления интерфейса без полной перезагрузки страницы.

## Архитектурная идея

Проект разделен на несколько логических слоев:

- `domain` — доменная модель, идентификаторы, value object, enum-значения и бизнесовые сущности.
- `storage/tables` — описание таблиц PostgreSQL через Exposed.
- `storage/repository` — чтение данных одной таблицы или базовые операции получения сущностей.
- `storage/persister` — операции создания и изменения данных.
- `storage/query` — отдельные запросы, которые объединяют несколько таблиц и возвращают DTO для конкретных сценариев.
- `service` — бизнес-логика, сборка доменных сущностей, валидация связей и координация операций.
- `controller` — HTTP endpoints.
- `controller/handler` — подготовка HTML-страниц и фрагментов интерфейса.
- `pages` — общие UI-компоненты, layout, таблицы и вспомогательная отрисовка.
- `security` — авторизация, роли и ограничения доступа.

Такой подход позволяет не смешивать SQL, бизнес-логику и HTML-отрисовку в одном месте. Например, создание записи на прием выполняется через сервис: сначала проверяется пациент, затем доступность временного слота, после этого создается запись и слот переводится в занятое состояние.

## База данных

Схема создается SQL-скриптами из директории `init-scripts` при первом запуске PostgreSQL-контейнера.

Основные таблицы:

| Таблица | Назначение |
| --- | --- |
| `specializations` | Справочник специализаций врачей. |
| `departments` | Отделения клиники. |
| `doctors` | Врачи, их контакты, лицензии, отделения и специализации. |
| `patients` | Пациенты, контактные данные, адрес, страховая информация. |
| `patient_medical_histories` | История болезни пациента. |
| `user_accounts` | Учетные записи пользователей и связь с врачом или пациентом. |
| `schedules` | Рабочие дни врачей. |
| `time_slots` | Временные слоты внутри расписания. |
| `appointments` | Записи пациентов на прием. |
| `SPRING_SESSION` | Сессии пользователей Spring Session JDBC. |
| `SPRING_SESSION_ATTRIBUTES` | Атрибуты пользовательских сессий. |

Также в схеме есть технические таблицы `outbox`, `inbox`, `change_log`, которые могут использоваться для расширения системы в сторону интеграций, аудита и событийной обработки.

## Роли пользователей

### Администратор

Администратор управляет справочниками и данными системы, видит общую картину по пациентам, врачам, расписанию и записям.

Возможности:

- доступ к дашборду;
- просмотр врачей и пациентов;
- создание и редактирование пациентов;
- редактирование врачей;
- создание записей на прием;
- импорт и экспорт данных пациентов.

### Врач

Врач работает только с теми данными, которые относятся к нему.

Возможности:

- просмотр собственных записей;
- просмотр связанных пациентов;
- просмотр расписания;
- добавление и редактирование записей истории болезни для своих пациентов.

### Пациент

Пациент видит только свою карточку и может самостоятельно записаться на прием.

Возможности:

- просмотр своей карточки;
- редактирование своих контактных данных;
- экспорт своих данных;
- просмотр истории болезни;
- самостоятельная запись на прием к врачу.

## Запуск через Docker Compose

Самый простой способ запустить проект — использовать Docker Compose.

```bash
docker compose up --build
```

После запуска приложение будет доступно по адресу:

```text
http://localhost:8080
```

PostgreSQL будет доступен на порту `5432`:

```text
jdbc:postgresql://localhost:5432/clinic
```

Параметры подключения по умолчанию:

```text
Database: clinic
User: clinic_user
Password: clinic_pass
```

## Остановка проекта

```bash
docker compose down
```

Если нужно полностью пересоздать базу и заново выполнить init-скрипты:

```bash
docker compose down -v
docker compose up --build
```

Важно: `docker compose down -v` удаляет volume PostgreSQL, поэтому все локальные данные базы будут потеряны.

## Локальный запуск без Docker-приложения

Можно поднять только PostgreSQL:

```bash
docker compose -f docker-compose-postgres.yml up -d
```

## Работа с CSV

В системе есть импорт и экспорт данных пациента.

Экспорт карточки пациента включает:

- персональные данные пациента;
- страховую информацию;
- связанные записи на прием;
- данные, необходимые для восстановления связей при импорте.

Отдельно поддерживается экспорт и импорт истории болезни пациента.

Файлы CSV формируются в кодировке UTF-8. Для совместимости с табличными редакторами при выгрузке добавляется BOM.

## HTMX и интерфейс

Интерфейс генерируется на сервере через `kotlinx.html`. Для частичных обновлений используется HTMX.

Примеры использования HTMX в проекте:

- поиск врача по ФИО без полной перезагрузки страницы;
- обновление фрагмента расписания при выборе врача или даты;
- обработка ошибок HTMX-запросов через общий toast-контейнер в `appLayout`.

Для визуальной части используется Bootstrap 5.

## Обработка ошибок

В проекте используется несколько уровней отображения ошибок:

- ошибки формы отображаются рядом с полем через Bootstrap `invalid-feedback`;
- общие ошибки формы выводятся через `alert alert-danger`;
- ошибки HTMX-запросов отображаются в правом нижнем углу через toast;
- ошибки доступа ведут на страницу `access-denied`.

Value object возвращают доменные сообщения об ошибках, чтобы пользователь видел понятное описание причины, а не техническое исключение.

## Предметная модель

В проекте используется доменная модель, где поля не передаются как примитивные строки и числа без смысла. Для важных бизнес-понятий используются value object и enum-значения.

Примеры:

- `Fio` — ФИО пациента или врача;
- `PhoneNumber` — номер телефона;
- `MedicalCardNumber` — номер медицинской карты;
- `AppointmentType` — тип приема;
- `AppointmentStatus` — статус записи;
- `TimeSlotStatus` — статус временного слота;
- `PatientGender` — пол пациента;
- `DoctorActivityStatus` — активность врача.

Это помогает держать бизнес-валидацию ближе к доменной модели и не размазывать правила по контроллерам.

## Repository, Persister, Service и Query

В проекте приняты следующие правила:

- `Repository` отвечает за чтение данных одной сущности или таблицы.
- `Persister` отвечает за создание и изменение данных.
- `Service` координирует бизнес-операции и работает с доменными сущностями.
- `Query` используется для сложных запросов с join нескольких таблиц и возвращает DTO конкретного сценария.

Например, список записей на прием строится отдельным query-классом, потому что для отображения нужны данные из таблиц пациентов, врачей, записей и слотов.

## Сессии пользователей

Сессии хранятся в базе данных через Spring Session JDBC. Это позволяет не терять авторизацию при перезапуске приложения, если база и volume PostgreSQL сохраняются.

Для этого используются таблицы:

- `SPRING_SESSION`;
- `SPRING_SESSION_ATTRIBUTES`.

Время жизни сессии настроено в `application.yml`:

```yaml
server:
  servlet:
    session:
      timeout: 12h
```

## Seed data

Файл `init-scripts/02_seed_data.sql` создает демонстрационные данные:

- специализации;
- отделения;
- врачей;
- пациентов;
- учетные записи пользователей;
- истории болезни;
- расписания;
- временные слоты;
- записи пациентов на прием.

Даты в seed-data формируются относительно `current_date`, поэтому демо-данные остаются актуальными при новом запуске.


## Назначение проекта

Проект демонстрирует, как можно построить небольшую медицинскую информационную систему без зависимости от внешних сервисов записи. Клиника получает собственную базу пациентов, врачей, расписаний, приемов и историй болезни, а накопленные данные могут использоваться для внутренней аналитики, оценки нагрузки и дальнейшего развития цифровых сервисов.
