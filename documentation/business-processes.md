# Бизнес‑процессы ИС записи пациентов (MVP)

## 1. Термины и роли
### Роли
- **Пациент**: создает/отменяет запись, получает уведомления.
- **Регистратор/Оператор**: создает запись от имени пациента, подтверждает/отменяет, управляет справочниками.
- **Врач**: просматривает свое расписание и список записей, может отмечать завершение приема (опционально).
- **Система уведомлений**: отправляет сообщения (SMS/Email/Push) и фиксирует статусы доставки.

### Основные сущности (в терминах процесса)
- Пациент, Врач, Расписание, Запись на прием, Уведомление.

---

## 2. Контекст и границы
### В рамках MVP
- Запись на прием к врачу по расписанию и слотам.
- Контроль конфликтов по времени (один слот — одна запись).
- Подтверждение/отмена записей.
- Уведомления о событиях (создано/подтверждено/отменено).

### Вне рамок (можно добавить позже)
- Оплата/страховые проверки.
- Электронная медкарта и результаты обследований.
- Интеграции с гос.реестрами, ЕМИАС и т. п.

---

## 3. Карта бизнес‑процессов (L0)
- **Управление справочниками** (пациенты, врачи).
- **Управление расписанием врача**.
- **Запись пациента на прием**.
- **Управление жизненным циклом записи** (подтверждение/отмена/завершение).
- **Уведомления и коммуникации**.

---

## 4. Процесс BP‑01: Регистрация/ведение пациента
### Цель
Создать или актуализировать карточку пациента для последующей записи на прием.

### Триггеры
- Новый пациент обращается впервые.
- Нужно обновить контактные данные.

### Предусловия
- Оператор или пациент имеет доступ к форме.

### Основной сценарий
1. Ввод ФИО и телефона (обязательные), email/дата рождения (опционально).
2. Система проверяет уникальность телефона (и email, если заполнен).
3. Система сохраняет пациента.
4. Возвращается идентификатор пациента.

### Исключения
- Телефон уже существует → предложить выбрать существующего пациента или обновить данные.
- Некорректный формат данных → отказ с сообщением об ошибке.

### Результат
Пациент создан/обновлен и доступен для записи.

---

## 5. Процесс BP‑02: Ведение врача
### Цель
Поддерживать актуальный справочник врачей и параметров приема.

### Основной сценарий
1. Создание/редактирование врача (ФИО, специальность, кабинет, лимиты).
2. (Опционально) управление доступностью (is_available).

### Результат
Врач доступен для построения расписаний и записей.

---

## 6. Процесс BP‑03: Управление расписанием врача
### Цель
Задать рабочие интервалы, перерывы и параметры слотирования.

### Основной сценарий
1. Оператор выбирает врача.
2. Задает период действия (date_from/date_to).
3. Задает рабочее время (start_time/end_time).
4. (Опционально) задает перерыв (break_start/break_end).
5. Задает длительность слота и лимит (slot_duration/max_slots).
6. Система сохраняет расписание.

### Контроли
- date_from ≤ date_to
- start_time < end_time
- break_start/break_end внутри рабочего интервала (если заданы)
- отсутствие пересечений расписаний (рекомендуемое правило)

---

## 7. Процесс BP‑04: Поиск доступных слотов
### Цель
Показать пациенту/оператору свободные времена для записи.

### Основной сценарий
1. Выбор врача и даты.
2. Система находит расписание на дату.
3. Генерирует сетку слотов по slot_duration.
4. Убирает слоты, попадающие в перерыв.
5. Убирает слоты, уже занятые активными записями.
6. Возвращает список свободных слотов.

### Исключения
- Нет расписания на дату → вернуть пустой список/сообщение.
- Врач недоступен → вернуть ошибку/пустой список.

---

## 8. Процесс BP‑05: Создание записи на прием
### Цель
Создать запись пациента к врачу на выбранный свободный слот.

### Участники
Пациент или Оператор.

### Предусловия
- Пациент и врач существуют.
- Расписание на дату задано.
- Слот свободен.

### Основной сценарий
1. Пользователь выбирает врача, дату, время и тип приема.
2. Система валидирует входные данные.
3. Система проверяет конфликт (слот уже занят?).
4. Система проверяет лимиты на дату.
5. Система создает запись (обычно со статусом PENDING или сразу CONFIRMED — по политике).
6. Система создает задачу/запись на уведомление.
7. Возвращает данные записи.

### Исключения
- Слот занят → отказ, предложить выбрать другое время.
- Лимит исчерпан → отказ, предложить другую дату.
- Ошибка сохранения → отказ, запись не создается (атомарность).

---

## 9. Процесс BP‑06: Подтверждение записи
### Цель
Перевести запись в подтвержденное состояние.

### Основной сценарий
1. Пользователь (оператор/пациент) инициирует подтверждение.
2. Система проверяет текущий статус (например, только PENDING).
3. Меняет статус на CONFIRMED, фиксирует время подтверждения.
4. Создает уведомление о подтверждении.

### Исключения
- Неверный статус → отказ (например, CANCELLED нельзя подтвердить).

---

## 10. Процесс BP‑07: Отмена записи
### Цель
Снять запись и освободить слот.

### Основной сценарий
1. Пользователь инициирует отмену.
2. Система меняет статус на CANCELLED, фиксирует время отмены.
3. (Опционально) сохраняет причину отмены.
4. Создает уведомление об отмене.

### Результат
Слот становится доступным для новой записи (если дата не прошла и это разрешено правилами).

---

## 11. Процесс BP‑08: Завершение приема (опционально)
### Цель
Отметить факт оказания услуги (для отчетности/истории).

### Основной сценарий
1. Врач или оператор выбирает запись.
2. Переводит в статус COMPLETED.
3. (Опционально) добавляет комментарий.

---

## 12. Процесс BP‑09: Отправка уведомлений
### Цель
Доставить пациенту сообщения о ключевых событиях и зафиксировать результат.

### Основной сценарий
1. Событие в ИС создает уведомление со статусом PENDING.
2. Воркер/планировщик отправляет через внешний канал (SMS/Email).
3. При успехе → SENT, фиксируется sent_at и delivery_time.
4. При ошибке → FAILED, фиксируется error_message (и возможны ретраи).

---

## 13. Набор бизнес‑правил (универсальный)
- BR‑01: Один врач не может иметь две активные записи на один и тот же слот.
- BR‑02: Запись возможна только внутри рабочего времени расписания и вне перерыва.
- BR‑03: Запись невозможна при отсутствии расписания на дату.
- BR‑04: Отмененная запись не считается занятостью слота.
- BR‑05: Подтверждение возможно только для PENDING (если такой статус используется).
- BR‑06: Лимит слотов на дату не должен превышаться.

---

## 14. Метрики (что полезно измерять)
- Количество записей по врачам/дням.
- Доля отмен.
- Среднее время до подтверждения.
- Доставка уведомлений: success rate, latency.
